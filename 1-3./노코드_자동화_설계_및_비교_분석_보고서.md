# 노코드 자동화 설계 및 비교 분석 보고서

본 보고서는 노코드 자동화 도구(Make, Zapier)를 활용한 워크플로우 구현, 핵심 개념 분석, 그리고 실무 적용을 위한 자유 주제 자동화 파이프라인 설계 내용을 담고 있습니다.

---

# [프로젝트 1] 자동화 도구 비교 구현 보고서

## 1. 개요 및 구현 과정 요약
*   **사용한 도구:** Make (구 Integromat), Zapier
*   **워크플로우 시나리오:** "구글 시트에 새로운 마케팅 리드(고객 정보)가 추가되면, 담당자에게 메일(Gmail)을 보내고, 이메일 주소가 없는 잘못된 데이터는 Slack으로 경고 메시지를 보낸다."
*   **구현 과정 요약:** 
    1.  **Trigger 설정:** Google Sheets의 'New Spreadsheet Row'를 시작점으로 지정하여 실시간 데이터 유입을 감지합니다.
    2.  **Router/Filter(조건 분기) 추가:** 데이터 중 '이메일' 열이 비어있는지, 정상적으로 채워져 있는지 판별하는 분기점을 생성합니다.
    3.  **Action 1 (정상 경로):** 이메일 주소가 존재하는 경우, Gmail 모듈을 통해 고객에게 자동으로 웰컴 메일을 발송합니다.
    4.  **Action 2 (오류 경로):** 이메일 주소가 누락된 경우, Slack 모듈을 통해 지정된 팀 채널로 "누락된 리드 발생, 수동 확인 필요" 경고 알림을 전송합니다.

---

## 2. 도구별 워크플로우 구성 및 실행 결과 (스크린샷 영역)
> ⚠️ **보안 및 제출 주의 사항:** 스크린샷을 첨부할 때는 API Key, 개인 이메일 주소, Webhook URL, 고유 토큰 값 등의 민감 정보가 노출되지 않도록 이미지 편집 툴을 활용하여 반드시 마스킹(`***`) 처리해야 합니다.

### ① Make (구 Integromat) 구현
*   **워크플로우 구성 화면 [캡처 가이드]:** 시각적인 원형 노드(Module)들이 중앙 Router를 중심으로 상하 분기되는 시나리오 전체 맵을 캡처하여 첨부합니다.
*   **실행 결과 화면 [캡처 가이드]:** 데이터가 유입되었을 때 각 노드 우측 상단에 초록색 체크표시($\checkmark$)와 함께 작업 처리 횟수(`1`)가 표시되며, 각 분기 경로가 정상적으로 구동된 로그 화면을 캡처합니다.

### ② Zapier 구현
*   **워크플로우 구성 화면 [캡처 가이드]:** 상단에서 하단으로 흐르는 계단식(순차적 트리 형태) 구조의 `Trigger -> Filter/Path -> Action` 전체 화면을 캡처하여 첨부합니다.
*   **실행 결과 화면 [캡처 가이드]:** Zap History 메뉴 진입 후, 두 가지 조건 분기 경로(Path A 및 Path B)가 입력 데이터의 조건에 따라 각각 작동하여 `Success` 상태로 처리된 상세 내역을 캡처합니다.

## 3. 핵심 개념 설명
*   **Trigger (시작 이벤트):** 자동화 워크플로우를 깨우는 '방아쇠' 역할을 합니다. "새 이메일 수신", "스프레드시트에 행 추가", "웹훅 요청 수신" 등이 이에 해당하며, 대개 실시간 통지(Webhook) 방식이나 주기적 조회(Polling) 방식으로 작동합니다.
*   **Action (처리 동작):** Trigger가 성공적으로 발생한 후 플랫폼이 사용자를 대신하여 수행하는 '실제 행동'입니다. "이메일 발송", "메시지 전송", "DB 업데이트" 등이 있으며, 각 서비스의 오픈 API 호출을 기반으로 작동합니다.
*   **조건 분기 (Filter/Router)의 역할:** 유입된 데이터를 검사하여 사용자가 지정한 특정 조건(예: `금액 > 100,000` 또는 `이메일 존재 여부`)을 만족하는 경우에만 특정 방향의 Action으로 흐르도록 제어하는 **데이터 신호등** 역할을 합니다. 이를 통해 하나의 트리거로 다양한 시나리오를 대응할 수 있습니다.

## 4. Make vs Zapier 상세 비교 분석

| 비교 항목 | Make (구 Integromat) | Zapier |
| :--- | :--- | :--- |
| **UI/UX & 시각화** | 2차원 캔버스 형태의 드래그 앤 드롭 공간을 제공합니다. 한눈에 복잡한 흐름과 다중 분기를 직관적으로 파악하기에 매우 유리합니다. | 1차원 수직/계단식 순차 구조를 가집니다. 단계별 흐름이 매우 깔끔하고 직관적이지만, 분기가 많아지면 가독성이 떨어집니다. |
| **설정 난이도** | 데이터 매핑 방식 자체는 직관적이나, 자체 내장 함수(Functions)나 데이터 파싱(Parsing) 규칙을 다룰 때 약간의 숙련도가 요구됩니다. | 초보자 친화적입니다. 직관적인 UI 가이드를 제공하며, 자연어에 가까운 안내 메시지를 따라가면 누구나 쉽게 연동을 마칠 수 있습니다. |
| **연동 서비스 범위** | 전 세계 메이저 서비스는 대부분 완벽히 지원하지만, 아주 지엽적이거나 마이너한 서드파티 앱의 지원 종류는 Zapier보다 적을 수 있습니다. | 현존 노코드 자동화 툴 중 독보적인 1위입니다. 6,000개 이상의 전 세계 다양한 웹 서비스 및 앱과의 연동을 기본 지원합니다. |
| **조건 분기 구현** | **Router** 노드를 활용하면 비용이나 플랜 제한 없이 3개 이상의 다중 분기 및 복잡한 조건 필터링을 매우 자유롭게 설계할 수 있습니다. | 무료 플랜에서는 다중 분기(Paths) 기능이 제한되거나 단일 필터만 허용되는 경우가 많아, 복잡한 로직 구현 시 유료 결제가 반강제됩니다. |
| **무료 플랜 범위** | 매월 **1,000번의 오퍼레이션(작업 단위)**을 기본 제공합니다. 여러 단계로 구성된 복잡한 시나리오도 무료 범위 내에서 충분히 테스트 가능합니다. | 매월 **100번의 태스크**만 제공됩니다. 특히 3단계 이상의 다단계(Multi-step) 자동화를 활성화하려면 유료 요금제 업그레이드가 필수적입니다. |
| **실행 로그 확인** | 각 모듈별 인풋/아웃풋 데이터 흐름(Data Balloon)을 시각적으로 즉시 열어볼 수 있어 디버깅 및 에러 추적이 매우 강력합니다. | 내역(Task History) 탭에서 텍스트 기반의 상세 로그를 제공하며, 단계별 데이터 전송 결과 및 에러 코드를 차례대로 확인해야 합니다. |

### 종합 의견 및 상황별 추천
*   **Make가 적합한 상황:** 자동화 예산을 최소화하면서 복잡한 다중 분기, 반복문(Iterator/Aggregator), 대량의 데이터 처리가 필요한 경우에 강력히 추천합니다. 데이터 흐름을 시각적으로 세밀하게 제어하려는 중·고급 사용자 및 개발자 지향 조직에 최적입니다.
*   **Zapier가 적합한 상황:** 자동화를 처음 도입하는 초보자나 비개발자 중심의 팀, 복잡한 로직보다는 생소한 서드파티 서비스들과의 빠르고 안정적인 연동이 최우선인 비즈니스 환경에 적합합니다. 100건 미만의 단순 일회성 태스크 자동화에 유리합니다.

# [프로젝트 2] 자유 주제 자동화 설계 및 구현 보고서

## 1. 반복 업무 정의
*   **업무 명칭:** AI 기반 고객 문의(CS) 자동 분류 및 비상 알림 시스템
*   **문제 상황:** 매일 구글 폼이나 스프레드시트를 통해 접수되는 고객 문의를 담당자가 일일이 수동으로 읽고 "환불 요구 / 단순 변심 / 시스템 오류" 등으로 분류한 뒤, 다시 담당 부서에 메신저로 전달하는 과정에서 매일 오전 평균 40~50분의 리소스가 낭비되고 있습니다. 특히 시스템 오류(버그)의 경우 조기 대응이 늦어지는 문제가 있습니다.
*   **자동화 목표:** 고객 문의가 접수되는 즉시 **생성형 AI(ChatGPT)가 문맥을 분석하여 카테고리를 자동 분류**하고, 긴급 처리가 필요한 '시스템 오류' 건에 대해서는 담당 개발팀 Slack 채널로 즉시 비상 알림을 전송하는 파이프라인을 구축합니다.

## 2. 도구 선정 및 선정 이유
*   **선정 도구:** **Make**
*   **선정 이유:** 본 워크플로우는 `시트 감지 -> OpenAI 분석 -> 조건 분기 -> 슬랙 전송 또는 시트 업데이트`로 이어지는 4단계 이상의 멀티 스텝(Multi-step) 구조입니다. Zapier 무료 플랜의 제약을 넘어 제한 없이 다단계 구조를 구현할 수 있고, AI가 반환한 텍스트 데이터를 정제하고 분기 조건(Filter)에 매핑하는 작업이 Make의 라우터 환경에서 훨씬 유연하고 직관적이기 때문입니다.

## 3. 워크플로우 설계 문서 (흐름도)

```
[Trigger: Google Sheets 새 행 추가] 
       │
       ▼
[Action 1: OpenAI ChatGPT 문맥 분석 및 카테고리 태깅] 
(고객 문의 내용을 프롬프트에 입력하여 "버그", "환불", "기타" 중 하나로 분류)
       │
       ▼
 [Router: 조건 분기 설정]
 ┌─────┴────────────────────────┐
 │ (조건: AI 분석 결과 == '버그')    │ (조건: AI 분석 결과 == '환불' 또는 '기타')
 ▼                              ▼
[Action 2-A: 개발팀 Slack 알림]  [Action 2-B: 구글 시트 상태 업데이트]
(개발 채널로 긴급 메시지 전송)      (해당 행의 처리 상태 열에 카테고리 기록)
```

## 4. 구현 및 동작 확인 (스크린샷 영역)
*   **구현 완료 화면 캡처 [가이드]:** Make의 전체 시나리오(구글 시트 -> OpenAI -> 라우터 -> 슬랙 & 구글시트 업데이트) 배치 스크린샷을 첨부합니다.
*   **실행 로그 및 결과 화면 캡처 [가이드]:** 
    *   **분기 경로 A 실행 결과:** 고객이 *"로그인 버튼을 눌러도 반응이 없고 하얀 화면만 뜹니다"*라고 입력했을 때, OpenAI가 이를 `버그`로 분류하여 Slack 개발 채널에 알림이 즉시 발송된 화면을 증빙합니다.
    *   **분기 경로 B 실행 결과:** 고객이 *"사이즈가 안 맞아서 환불하고 싶어요"*라고 입력했을 때, OpenAI가 이를 `환불`로 분류하여 슬랙 알림 없이 구글 시트의 '분류' 열에 정상 기록된 결과를 증빙합니다.

# 🚀 보너스 과제 구현 가이드 (선택 사항)

### 1. AI 연동 Action 추가 방법 (OpenAI 프롬프트 엔지니어링 팁)
워크플로우 중간에 OpenAI 모듈의 `Create a Completion` 또는 `Create a Chat Completion`을 배치할 때, 조건 분기가 오작동하지 않도록 **출력 값을 고정**하는 프롬프트 작성이 핵심입니다.
*   **추천 프롬프트 구조:**
    ```text
    너는 입력된 고객 문의를 정확하게 분류하는 분류 에이전트야.
    다음 기준에 따라 오직 한 단어('버그', '환불', '기타')로만 응답해줘. 
    불필요한 미사여구, 마침표, 설명은 절대로 생략하고 단어만 출력해.

    [기준]
    - 시스템 오류, 로그인 불가, 페이지 깨짐 -> 버그
    - 반품, 취소, 환불 요청 -> 환불
    - 그 외 단순 문의 및 인사 -> 기타

    [고객 문의 내용]
    {{1. Google Sheets: 문의내용}}
    ```

### 2. 실패 알림 및 에러 핸들링 (재시도 및 대체 경로 설계)
네트워크 불안정이나 API 할당량 초과 등으로 워크플로우가 멈추는 것을 방지하기 위해 Make의 에러 핸들러(Error Handler) 기능을 적용합니다.
1. 에러 발생 확률이 높은 외부 연동 모듈(예: Gmail, Slack)을 마우스 우클릭한 후 **`Add error handler`**를 선택합니다.
2. 다양한 에러 핸들러 지시어 중 상황에 맞는 동작을 연결합니다:
    *   **`Retry`:** 일정 시간 간격을 두고 자동으로 재실행을 시도합니다.
    *   **`Resume` / `Ignore`:** 에러를 무시하거나 기본 대체 값을 삽입하여 다음 프로세스가 멈추지 않고 진행되도록 합니다.
3. 에러 핸들러 노드 뒤에 **Slack/이메일 알림 모듈**을 추가로 연결하여, *"자동화 시나리오 X단계에서 [Error 메시지]가 발생하여 우회 처리되었습니다. 확인 바랍니다."* 형태의 관리자 알림 시스템을 구축하면 운영 안정성을 극대화할 수 있습니다.
